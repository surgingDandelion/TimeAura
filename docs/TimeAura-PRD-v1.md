# TimeAura 产品需求文档（PRD v1.2）

## 1. 文档信息

- 产品名称：TimeAura
- Slogan：让每个重要时刻如约而至
- 产品形态：AI 备忘录与待办桌面应用
- 版本阶段：V1.2 设计输入稿 + 交互状态规范版
- 文档日期：2026-03-24
- 目标平台：桌面端优先（macOS / Windows，跨平台框架可扩展）

## 2. 产品愿景

TimeAura 是一款面向知识工作者、个人管理者与轻量团队用户的桌面备忘录应用。它将“快速记录”“结构化整理”“时间驱动提醒”和“AI 智能总结”融合为一个统一工作台，帮助用户以更低的输入成本捕捉任务、沉淀信息，并在周报、月报、复盘与提醒等关键时刻自动生成高价值输出。

TimeAura 不只是记事工具，而是一个以“时间”为主线、以“重要时刻”为交付结果的智能个人事务中枢。

## 3. 产品定位

### 3.1 一句话定位

一个专注于“待办 + 备忘 + AI 总结”的桌面效率应用，帮助用户从碎片记录走向按时间兑现结果。

### 3.2 核心价值

- 更快记录：支持单行快速新增、快捷完成时间设置、低打扰输入。
- 更好整理：支持标签、优先级、状态、排序、模糊检索与单行列表浏览。
- 更强表达：支持 Markdown 编辑与预览，兼顾轻量记录与结构化内容创作。
- 更智能产出：支持 AI 通道配置、Prompt 配置、报告模板配置，自动生成周报、月报、阶段复盘。
- 更贴近时间：围绕今天、即将到期、已完成、逾期等时间视角组织信息。

## 4. 目标用户

### 4.1 核心用户

- 个人效率用户：希望把待办和笔记放在一个应用里管理。
- 知识工作者：需要持续记录工作事项，并在固定周期输出周报、月报。
- 管理岗 / 项目负责人：需要从日常记录中汇总事项进展、风险、完成度。
- AI 重度用户：有多个模型或 API 渠道，希望自行配置和切换 AI 能力。

### 4.2 典型使用场景

- 场景 A：随手输入一句话，快速生成一条待办，并设置“今天下班前完成”。
- 场景 B：为备忘录补充 Markdown 内容、标签与优先级，沉淀过程信息。
- 场景 C：通过模糊搜索快速找到“上周关于招聘渠道优化”的所有记录。
- 场景 D：基于本周已完成事项、延期事项、重点标签，自动生成周报草稿。
- 场景 E：切换不同 AI 通道，分别用于“摘要生成”“结构化提炼”“长文润色”。

## 5. 产品目标

### 5.1 业务目标

- 建立一款差异化明显的桌面效率应用原型。
- 在“记录-整理-产出”链路中，用 AI 显著提升用户产出效率。
- 通过 AI 配置开放能力，为后续高级版/专业版打基础。

### 5.2 用户目标

- 用户可以在 3 秒内完成一条待办录入。
- 用户可以在 10 秒内找到一条已有备忘录。
- 用户可以在 1 分钟内生成结构清晰、可编辑的周报草稿。

## 6. 产品原则

- 低输入门槛：优先保证“快速记下”而不是强制完整填写。
- 列表高密度：主列表采用单行信息组织，快速扫视、快速操作。
- 细节可渐进补全：先创建，后补标签、日期、内容、AI处理。
- AI 可控：所有 AI 功能支持模型、Prompt、输出格式和通道配置。
- 时间导向：围绕截止时间、计划时间、完成时间构建关键体验。

## 8. 信息架构

应用建议采用“三栏主体 + 左下角功能按钮”的简约桌面结构，参考 mac 风格侧栏设计，减少功能堆叠，突出时间状态与标签入口。

### 8.1 左侧导航区

- App 名称与品牌区
- 今天
- 计划
- 全部
- 已完成
- 标签列表（标签与分类合并后的统一体系，直接展示，不单独做“标签功能区”标题）
- 每个系统视图与标签右侧展示待办数字
- 左下角固定图标按钮：AI 报告、回收站、设置

### 8.2 中间主列表区

- 顶部快速新增输入框
- 搜索框
- 筛选器：状态、优先级、标签、时间范围
- 排序器：创建时间、更新时间、截止时间、优先级、完成时间
- 即将过期任务快捷调整条
- 单行列表项

### 8.3 右侧详情区

- 标题
- 状态
- 优先级
- 标签
- 日期时间
- Markdown 内容编辑器
- 预览切换
- AI 操作区
- 元信息区（创建时间、更新时间、完成时间）
- 默认收起，用户点击列表项后再展开
- 与中间列表之间支持拖拽调节宽度
- 支持主动收起，收起后中间列表自动扩展

## 9. 核心概念定义

### 9.1 记录类型

V1 明确将“备忘录”和“待办”合并为统一 Record 模型，所有内容都在同一个列表视图中管理，通过字段差异体现使用方式。

- 轻量备忘：可无截止时间、无完成动作
- 待办事项：具备状态流转、计划时间、完成时间
- 长文备忘：正文内容较长，以 Markdown 为主

### 9.2 状态定义

建议 V1 采用以下状态：

- `未开始`
- `进行中`
- `已完成`
- `已归档`

可选扩展：

- `已搁置`
- `已取消`

说明：如果 V1 追求简洁，列表筛选层可展示为“全部 / 未完成 / 已完成”，而详情层保留更细粒度状态。

### 9.3 优先级定义

建议 4 级：

- `P1 紧急重要`：红色
- `P2 重要`：橙色
- `P3 常规`：蓝色
- `P4 低优先级`：灰色

说明：颜色需同时出现在列表项、详情区和筛选器中，形成统一识别。

### 9.4 标签定义

- 一个记录可绑定多个标签
- 标签支持颜色
- 标签支持模糊检索
- 标签支持快速创建
- 标签与分类合并为同一套体系
- 左侧侧栏直接展示标签列表及数量，不额外拆出标签管理区域

### 9.5 系统视图定义

为保证左侧导航、角标数字、提醒逻辑和排序逻辑一致，V1 明确采用以下系统视图定义：

- `今天`
  - 展示所有未完成且满足以下任一条件的记录：
    - `due_at` 在今天
    - `planned_at` 在今天
    - 已逾期未完成
  - 列表内默认按“已逾期 -> 今天即将到期 -> 其他今天事项”分组排序
- `计划`
  - 展示所有未完成，且 `due_at` 或 `planned_at` 晚于今天 23:59 的记录
  - 不包含无时间记录
- `全部`
  - 展示所有未归档、未删除、可见于主列表的非报告记录
  - 包括无时间记录
- `已完成`
  - 展示状态为 `已完成` 的记录
  - 默认按 `completed_at` 倒序排列

### 9.6 角标数量规则

- 左侧系统视图角标默认显示“未完成记录数”
- `已完成` 视图角标显示“已完成记录数”
- 标签角标显示该标签下“未完成记录数”
- 报告记录默认不计入 `今天 / 计划 / 全部` 的角标

### 9.7 AI 报告与记录关系

为避免 AI 生成结果成为孤立内容，V1 采用以下规则：

- AI 生成的周报 / 月报在“生成中”阶段只是临时草稿
- 用户点击“保存为报告记录”后，系统创建一条特殊的 `ReportRecord`
- `ReportRecord` 属于统一 Record 体系，但默认不展示在主备忘录列表中
- `ReportRecord` 主要展示在 `AI 报告` 页面下的历史记录区域
- 用户可手动选择“加入主列表”，此时该报告可出现在 `全部` 视图中

说明：这样既保持统一数据模型，又不污染主任务列表。

## 10. 核心功能需求

## 10.1 快速新增

### 目标

让用户无需打开详情面板，即可通过单行文本快速创建记录。

### 交互要求

- 顶部输入框常驻
- 窗口右上角提供 `快速新增` 入口，点击后必须切回主工作台、聚焦顶部输入框，并给输入框一段短暂高亮反馈
- 输入回车后立即创建记录
- 支持自然语言时间识别，如：
  - `明天下午 3 点和设计评审`
  - `周五前提交发票`
  - `月底整理月报素材`
- 支持快捷语法增强输入，如：
  - `#工作`
  - `!P1`
  - `@今天 18:00`
- 新增后自动聚焦下一条输入，适合连续录入
- 若当前不在主工作台页，点击 `快速新增` 后不得无反馈停留在当前页

### 默认规则

- 未指定优先级时默认为 `P3`
- 未指定状态时默认为 `未开始`
- 未指定标签时进入当前视图上下文
- 未指定时间时默认为“无截止时间”

## 10.2 单行列表展示

### 目标

让列表具备“高扫视效率”和“低认知负担”。

### 列表项字段

- 完成勾选框
- 标题（单行省略）
- 优先级色标
- 标签
- 状态
- 截止时间 / 计划时间
- AI 摘要标识（可选）
- 是否逾期标记

### 交互要求

- 单击选中并在右侧打开详情
- 双击进入标题快速编辑
- 勾选即完成
- 即将过期或已逾期任务支持直接在列表中一键调整时间
- 悬停显示快捷操作：选中、改期、AI 总结（V1 原型中至少落地选中与改期）
- 右侧详情默认不展示，只有当用户点击某条记录时才出现
- 详情展开后支持通过中间分割条鼠标拖拽调整宽度
- 列表右侧信息区需采用稳定对齐结构，建议为“时间信息 + 优先级标记”右对齐，不允许优先级漂浮在标题区附近造成视觉散乱
- 点击 `选中` 后必须立即出现三层反馈：
  - 当前行进入选中态
  - 按钮文案切换为 `已选中`
  - 底部批量操作条展示当前选中数量
- 若用户取消选中，上述反馈需同步回退

## 10.3 标签管理

### 功能点

- 标签新增、编辑、删除
- 标签颜色设置
- 标签下记录数展示
- 支持将标签作为左侧侧栏分组入口
- 支持在详情区和快速输入中添加标签
- 标签的新增、编辑、删除与清理入口放在右侧详情中的“管理标签”能力内，不放在左侧导航 hover 操作里
- 默认标签 `未分类` 不允许删除
- 删除标签时：
  - 若记录仍有其他标签，则仅移除当前标签
  - 若记录只绑定该标签，则自动回落到 `未分类`
  - 若当前筛选或当前侧栏选中的是被删除标签，需自动退回 `全部`

## 10.4 优先级管理

### 功能点

- 4 级优先级
- 列表中以色点 / 色条展示
- 支持按优先级筛选与排序
- 支持快速修改优先级

### 设计建议

- P1：#FF5A5F
- P2：#FF9F43
- P3：#4D8BFF
- P4：#98A2B3

## 10.5 状态管理

### 功能点

- 支持状态筛选
- 支持状态切换
- 已完成记录自动记录完成时间
- 支持“完成后归档”规则开关

## 10.6 排序与筛选

### 筛选维度

- 全部 / 未完成 / 已完成
- 优先级
- 标签
- 时间范围
- 是否逾期
- 是否含 AI 生成内容

### 排序维度

- 创建时间
- 更新时间
- 截止时间
- 完成时间
- 优先级
- 标题拼音 / 字母序（可选）

### 排序规则建议

- 默认视图：按截止时间优先，其次优先级，再次更新时间倒序
- 已完成视图：按完成时间倒序
- 无截止时间事项：排在有明确时间事项之后
- 即将到期且高优先级任务在同级排序中优先置顶

## 10.7 模糊检索

### 检索范围

- 标题
- 正文内容
- 标签
- 状态
- 时间字段文本

### 能力要求

- 支持中文模糊匹配
- 支持拼音首字母匹配（建议作为增强项）
- 支持高亮匹配结果
- 搜索实时返回

## 10.8 Markdown 编辑与预览

### 功能点

- 支持标准 Markdown 语法
- 支持编辑 / 分栏 / 预览三种模式
- 支持标题、列表、引用、代码块、分割线、粗体、斜体
- 支持复制渲染结果

### V1 建议

- 优先保证 Markdown 基础语法与稳定预览
- 图片粘贴上传、附件管理可以放到 V1.5

## 10.9 时间与快捷完成配置

这是 TimeAura 的差异化关键点之一，建议重点强化。

### 功能目标

在“完成待办”动作发生时，允许用户快速补充完成时间语义，而不是只有一个机械的“现在完成”。

### 快捷完成选项

- 现在完成
- 今天上午完成
- 今天下午完成
- 今天下班前完成
- 昨天完成
- 指定时间完成

### 应用场景

- 用户补录昨天完成的事项
- 用户需要让周报统计更准确
- 用户希望完成数据与实际工作节奏一致

### 交互建议

- 勾选完成后弹出轻量时间选择气泡
- 默认 1 秒内不操作则按“现在完成”
- 支持在设置中关闭弹窗，改为直接完成

## 10.10 提醒通知与时间调整

这是 V1 必须进入的能力，用来体现 TimeAura 的“时间感”与任务调度价值。

### 功能目标

- 对即将过期任务给出明确提醒
- 当多个任务积压时，主动提示用户调整时间或重新排序
- 让用户可以一键顺延、改期，而不是逐条编辑

### 提醒触发场景

- 距离截止时间 24 小时内
- 距离截止时间 2 小时内
- 已逾期未完成
- 同一时间段内积压多条高优先级任务

### 提醒内容建议

- 即将过期：提醒“今天还有 3 条重要事项待完成”
- 已逾期：提醒“2 条任务已过期，建议立即处理或调整时间”
- 积压冲突：提醒“今晚 18:00 前有 4 条任务重叠，建议重新安排”

### 快捷调整能力

- 延后 1 小时
- 延后到今天下班前
- 延后到明天上午
- 延后到本周五
- 自定义时间
- 批量调整选中任务时间
- 展开查看本次提醒命中的任务
- 仅对命中任务中的选中部分执行一键改期

### 交互建议

- 在列表顶部展示“即将过期任务条”
- 在任务行悬停时展示“一键改期”
- 提醒条支持展开命中任务清单，并显示“已选数量 / 命中数量”
- 在积压场景下弹出轻量提醒卡片，而不是强打断式弹窗
- 系统提醒支持静默模式与免打扰时间段

### 默认通知策略

V1 为避免打扰过度，默认采用“应用内提醒优先，系统通知精简触发”的双层策略：

- 应用内提醒
  - 只要用户打开应用，若存在即将到期、逾期或积压任务，则在主列表顶部展示提醒条
  - 应用内提醒不限制展示次数，但同类提醒文案需合并
- 系统通知
  - 仅对存在 `due_at` 的未完成任务生效
  - `P1 / P2` 默认在到期前 24 小时、到期前 2 小时、到期时各提醒一次
  - `P3 / P4` 默认在到期前 2 小时、到期时各提醒一次
  - 当天存在 3 条及以上未完成且时间重叠的任务时，在本地时间 18:00 发送一次积压摘要提醒

### 通知去重与合并规则

- 同一条任务在 30 分钟内不重复发送同等级提醒
- 同一时间窗口内的多条任务优先合并为一条摘要通知
- 若用户已在应用内处理提醒条，则同批次系统通知自动取消
- 免打扰时间段默认关闭系统通知，仅保留应用内提醒

### 通知快捷操作

桌面系统通知卡片建议支持以下快捷动作：

- `完成`
- `30 分钟后提醒`
- `改到明天上午`
- `打开详情`

### 批量改期规则

为避免改期结果不可预测，V1 明确以下规则：

- `延后 1 小时`
  - 若记录存在 `due_at`，则以该记录当前 `due_at + 1 小时` 为新时间
  - 若记录没有 `due_at`，则以“当前时间 + 1 小时”作为新时间
- `延后到今天下班前`
  - 固定设置为用户本地时区的 `今天 18:00`
  - 若当前时间已晚于今天 18:00，则该快捷项隐藏
- `延后到明天上午`
  - 固定设置为用户本地时区的 `明天 09:00`
- `延后到本周五`
  - 固定设置为用户本地时区的 `本周五 18:00`
  - 若今天已超过本周五，则指向下周五 18:00
- `自定义时间`
  - 打开轻量日期时间选择面板

### 批量改期边界规则

- 仅未完成记录可参与批量改期
- 已完成、已归档、已删除记录不可直接改期
- 从提醒条展开后，默认只允许操作当前提醒命中的任务
- 若用户勾选了部分任务，则改期仅作用于选中部分
- 若用户未勾选任何任务，则 `一键调整` 默认作用于全部命中任务
- 改期后任务需即时重排，并重新计算提醒状态

### 通道失败回退策略

为保证 AI 能力可用性，V1 明确采用以下回退策略：

- 每个 AI 能力优先使用其绑定通道
- 若绑定通道被禁用、测试失败或请求超时，则优先重试 1 次
- 重试失败后：
  - 若已开启“允许自动回退”，则自动切换到默认通道继续执行
  - 若未开启“允许自动回退”，则提示用户手动切换通道
- 若默认通道也失败，则保留当前输入条件、模板与草稿内容，并展示可重试错误态
- UI 需明确告知本次是否发生了自动回退，避免用户误解输出来源

## 11. 关键流程

## 11.1 快速新增流程

1. 用户在顶部输入框输入一行文本
2. 系统解析标题、标签、时间、优先级等结构化信息
3. 创建记录并展示到当前列表
4. 若解析失败，不阻塞创建，仅保留原始文本

## 11.2 完成待办流程

1. 用户勾选列表项
2. 系统弹出快捷完成时间面板
3. 用户选择时间或忽略
4. 系统写入完成状态与完成时间
5. 如开启“完成后归档”，自动移入已完成视图

## 11.3 即将过期任务调整流程

1. 系统识别 24 小时内即将到期或已逾期任务
2. 在主列表顶部展示提醒条和待调整数量
3. 用户可点击“展开命中任务”，查看本次提醒命中的全部事项
4. 用户可全选，也可只勾选其中一部分任务
5. 用户点击“一键调整”或某个快捷改期动作
6. 系统弹出快捷改期面板，作用于当前选中任务；若未选中，则默认作用于全部命中任务
7. 调整后任务列表即时重排，并更新提醒状态

## 11.4 周报生成流程

1. 用户进入 AI 报告页
2. 选择报告类型“周报”
3. 框选时间范围，并可附加标签范围、状态范围
4. 选择 AI 通道与模板
5. 系统生成 Markdown 报告
6. 用户编辑、保存、复制或导出

## 11.5 AI 通道调用流程

1. 用户在 AI 报告页或详情区触发 AI 能力
2. 系统读取该能力绑定通道
3. 若绑定通道可用，则直接执行
4. 若绑定通道不可用，则按“重试一次 -> 自动回退到默认通道”的规则处理
5. 系统返回生成结果，并标记使用通道
6. 若用户选择保存，则创建对应的 `ReportRecord` 或写回当前 Record

## 12.3 页面级 UI 设计约束

以下内容用于直接指导 UI 设计模型产出高保真效果图。

### 页面 A：主工作台 UI 约束

- 必须保留三栏结构：左侧导航、中间列表、右侧详情
- 左侧栏宽度建议 220-260px，背景比主区略深一级
- 系统入口仅保留 `今天 / 计划 / 全部 / 已完成`
- 标签列表紧接系统入口下方，不要再做单独大标题分组
- 左下角固定纯图标按钮：`AI 报告 / 回收站 / 设置`
- 顶部快速新增输入框必须强可见、长条形、可连续输入
- 提醒条必须处于筛选区下方、列表上方，视觉优先级高于普通列表项
- 单行列表项高度建议 44-56px
- 右侧详情编辑器要明显比左侧更舒展，适合长文编辑
- 右侧详情默认隐藏，用户点击任务后再展示
- 详情区默认宽度建议 520-560px，并允许拖拽调整到更宽
- 中间与右侧的分隔条应具备明确的拖拽暗示
- 详情中的日期时间输入框字体和控件高度应低于标题层级，避免喧宾夺主

### 页面 B：AI 报告页 UI 约束

- 页面必须与主工作台明显区隔，形成“生成工作台”氛围
- 左侧是条件配置面板，右侧是生成结果编辑区
- `生成报告` 按钮应是左侧主按钮
- `模板` 与 `AI 通道` 必须是相邻配置项，便于用户理解“模板 + 模型”的组合关系
- `模板配置` 入口应紧邻 `模板` 选择器，形成可发现关系
- 右侧结果区默认以 Markdown 富文本效果展示
- 顶部操作至少包含：`重新生成 / 复制 / 保存为记录 / 导出 Markdown`

### 页面 C：AI 通道配置页 UI 约束

- 必须采用左右分栏：左侧列表，右侧表单
- 左侧列表项要明确展示通道名称、服务商类型、启用状态
- `测试连接` 按钮必须显著
- `能力映射` 建议以紧凑表单或表格样式呈现
- 高级参数应可折叠，默认不要全部展开
- 服务商类型中需明确支持 `Anthropic 协议`

### 页面 E：主题切换 UI 约束

- 产品必须支持浅色主题与暗色主题
- 主题切换入口建议放在窗口顶栏或设置页中
- 浅色主题保持当前克制、干净、轻专业方向
- 暗色主题不走霓虹赛博路线，应保持冷静、细腻、桌面原生感
- 明暗主题切换时，优先保持布局、层级与交互一致，不改变信息结构

### 页面 D：提醒条与批量改期 UI 约束

- 提醒条应更接近轻量横幅，而不是错误告警条
- 展开命中任务后，必须能清晰看到 `已选数量 / 命中数量`
- `仅改选中` 必须是独立按钮，避免和“全部顺延”混淆
- 批量改期弹层需轻量，不做完整日历调度器
- 时间快捷选项优先级从左到右依次降低：立即可用快捷项 > 自定义时间
- 暗色主题下，改期弹层中的任务名称、时间文本、快捷选项说明必须与背景保持 AA 级对比，不允许出现浅色文字叠加浅色底导致不可读

## 12.4 交互状态规范

以下规范用于直接约束高保真 UI 与前端实现，避免“只画出静态页面，没有真实桌面应用反馈”。

### 12.4.1 全局交互反馈

- `hover`
  - 所有可点击按钮、列表项、标签项、选择项在悬停时必须出现至少一种反馈：背景变化、边框增强、轻微阴影或位移
  - hover 反馈强度要克制，不能出现过度放大、炫光或明显跳动
- `focus`
  - 输入框、下拉框、按钮、可切换项必须有清晰 `focus-visible` 状态
  - 焦点环建议使用低饱和蓝色描边或外发光，保持 mac 原生工具气质
- `active / pressed`
  - 按钮按下时应有轻微压下反馈
  - 不允许 hover 与按下态完全一致，否则会削弱操作确认感
- `selected`
  - 侧栏当前页、当前标签、当前 AI 通道、当前报告历史、当前任务多选，都必须有稳定选中态
  - 选中态优先通过背景层、描边和文字颜色联合表达，不依赖单一颜色
- `disabled`
  - 禁用态必须同时满足：不可点击、颜色降低、阴影取消、光标变更
  - 禁用态与普通次级按钮的差异要足够明显，避免误判为可用

### 12.4.2 快速新增状态规范

- 顶部快速新增输入框为主工作台的一级入口，始终高可见
- 点击窗口右上角 `快速新增`：
  - 若当前不在主工作台，先切回主工作台
  - 自动聚焦输入框
  - 输入框内容若存在则全选
  - 外层卡片出现 1 次短暂高亮，明确告知用户已进入录入态
- 输入框 `focus` 时：
  - 外层容器边框增强
  - 输入区背景略提升亮度
- 输入回车成功后：
  - 新记录立即插入当前列表
  - 输入框清空并保留焦点
  - 给出轻量 toast 或状态反馈

### 12.4.3 主列表行状态规范

- 默认态
  - 每行保持单行高密度
  - 右侧信息区采用固定结构：`时间` 在左、`优先级` 在右，整体右对齐
- hover 态
  - 行背景略抬升
  - 边框或阴影轻微增强
  - 行内快捷操作显示
- active 态（打开详情）
  - 当前行需明显区别于普通 hover
  - 建议使用更稳定的底色 + 描边，表示该行正在右侧详情区编辑
- selected 态（用于批量操作）
  - 当前行出现选中色底
  - `选中` 按钮变为 `已选中`
  - 按钮自身也进入高亮态
- done 态
  - 标题允许删除线
  - 时间和状态可弱化，但不得影响可读性

### 12.4.3A 详情区标签管理规范

- 左侧标签项默认只展示名称与数量，不承载编辑 / 删除按钮，保持侧栏纯净
- 标签维护入口放在右侧详情区的 `管理标签` 按钮中
- 打开标签管理后，至少支持：
  - 标签名称
  - 标签颜色
- 删除标签前必须二次确认，并明确告知对既有记录的影响
- `未分类` 作为系统兜底标签，允许编辑名称样式需谨慎，默认不允许删除
- 标签管理弹窗中应同时支持两类动作：
  - 维护标签库（新增 / 编辑 / 删除）
  - 调整当前记录绑定的标签集合

### 12.4.4 提醒条与批量改期状态规范

- 提醒条默认态
  - 展示提醒标题、说明文案、快捷改期入口
- 展开态
  - 展示命中任务列表
  - 明确显示 `已选 / 命中` 数量
- 命中项勾选态
  - 整行高亮，不只变化 checkbox
  - 保证用户能快速扫视哪些任务将被改期
- `仅改选中`
  - 当未选择任何命中项时必须禁用
- `一键调整`
  - 若已勾选命中项，则只作用于已勾选项
  - 若未勾选命中项，则默认作用于全部命中项，并在文案或反馈中说清楚
- 自定义改期弹层
  - 选择 `自定义时间` 前，时间输入框默认禁用
  - 选择 `自定义时间` 后，输入框启用
  - 未填写时间时，确认按钮禁用

### 12.4.5 暗色主题可读性规范

- 暗色主题不是简单反色，而是需要保持原生桌面工具的冷静层次
- 暗色下以下文本必须逐项校验对比度：
  - 任务标题
  - 次级说明
  - 改期弹层任务名与时间
  - 选项说明文字
  - 输入框占位符
- 原则：
  - 主文本使用高亮浅色
  - 次文本使用低一级灰蓝
  - 禁止在浅灰底上继续使用近白文字
  - 禁止使用亮蓝荧光色作为大面积正文文字

### 12.4.6 滚动条与桌面感细节规范

- 主列表、报告历史、报告编辑区、弹层列表都应使用统一风格滚动条
- 滚动条默认低存在感，hover 时增强
- 分割条 hover 与拖拽时必须强化视觉提示，明确其可拖动属性
- `Esc` 优先关闭当前弹层，其次收起右侧详情区
- 顶层按钮、列表与表单的动画时长建议控制在 `160ms - 220ms`

### 12.4.7 Design Tokens 设计输入规范

以下内容用于直接喂给 UI 设计模型、视觉设计师或前端实现，目标是让输出更接近“可落地的 mac 专业桌面工具”，而不是概念化设计稿。

#### A. 使用原则

- 所有页面优先使用“语义 token”而不是自由发挥颜色
- 浅色与暗色主题必须共用同一套结构、间距、层级、状态逻辑
- 未在 token 中明确声明的部分，优先向“中性、克制、低饱和”靠拢，不允许自行引入紫色主视觉、赛博蓝、过强玻璃拟态
- 蓝色只作为“焦点、选中、主要确认动作”的辅助语义，不作为大面积底色
- 橙 / 红 / 绿只用于时间状态、风险、完成、提醒，不得扩张为大面积装饰色
- 所有控件与面板默认应有轻微边界、低强度阴影、细腻高光，模拟 mac 原生工具的实装质感

#### K. 页面级组件映射

##### K1. 左侧栏

- 背景比主区略深半级
- 当前选中项：浅底 + 细描边 + 极弱阴影
- hover 只允许 `translateX(1px)` 或同等级轻反馈
- 不允许在左侧直接叠加重操作按钮，保持纯净导航感

##### K2. 主列表

- 列表行永远保持“单行信息组织”
- hover 不要做成卡片弹出，只能轻抬升
- active 态比 hover 更稳，selected 态比 active 更明确
- 右侧信息区固定右对齐：时间在左、优先级在右

##### K3. 详情 Inspector

- 必须像专业工具属性面板，而不是通用后台表单
- 字段按“分组 + 字段行”组织
- 分隔线要轻，组间距要稳定
- 标题、字段 label、字段值要有三级层次，不允许同权重挤在一起

##### K4. AI 报告页

- 左侧条件区更像控制台，右侧输出区更像文稿面板
- Banner、模板、通道、生成按钮需要形成“生成链路感”
- 右侧 Markdown 预览必须保留文稿阅读感，不要做成普通 textarea

#### L. 动效 Token

| Token | 值 | 用途 |
| --- | --- | --- |
| `motion.fast` | `160ms` | hover / pressed |
| `motion.base` | `180ms` | 输入框、按钮、chip |
| `motion.panel` | `220ms` | 详情展开、弹层浮现、分栏切换 |
| `motion.easing` | `ease` 或近似 mac 的平滑缓动 | 通用交互 |

规则：

- 不允许使用夸张弹跳、回弹过强、持续闪烁
- 动画是帮助确认交互，不是装饰

#### M. 禁止项

- 禁止紫色主视觉、荧光蓝、大面积高饱和科技渐变
- 禁止把主列表做成营销卡片墙
- 禁止大面积黑底高对比蓝光按钮
- 禁止所有按钮都带强阴影
- 禁止输入框字号大于详情标题
- 禁止 hover 与 selected 颜色过于接近，导致用户无法区分

#### N. 可直接喂给 UI 模型的摘要指令

若后续需要把 PRD 直接输入到 UI 设计模型，可附加以下摘要约束：

> 为 TimeAura 设计桌面端高保真 UI。风格接近 mac 原生专业工具，克制、冷静、轻专业，不是后台系统，不是营销页，不要赛博科技风，不要紫色主视觉。使用三栏结构：左侧简洁导航、中间高密度单行任务列表、右侧 inspector 属性面板。整体使用低饱和时间蓝、暖橙提醒、柔和红色风险、自然绿色完成。浅色与暗色主题共用同一信息架构。控件需原生化：select、date、checkbox、segmented 要像真实桌面产品，而不是设计稿装饰控件。主列表 hover 轻、selected 稳、active 清晰。滚动条、分隔线、阴影、圆角统一且克制。

## 14. 非功能需求

### 性能

- 主列表打开时间 < 1.5 秒
- 搜索响应时间 < 300ms
- 快速新增操作响应 < 100ms

### 安全

- API Key 本地加密
- 支持应用锁定（V1.5 可加）
- 日志中不得明文记录密钥

### 可用性

- 支持快捷键
- 支持键盘优先操作
- 支持浅色 / 暗色主题切换
- 系统通知支持可配置开关与免打扰

### 稳定性

- AI 调用失败不影响基础记事能力
- 本地数据保存需防止异常退出丢失
- 提醒失败不应影响任务数据本身

## 18. 风险与难点

- 自然语言时间解析在中文语境下需要兼顾准确率与容错率
- AI 输出质量高度依赖 Prompt、模板和输入数据结构
- 桌面端本地存储与密钥安全需要优先设计
- 列表高密度展示下，信息多而不乱的视觉平衡需要重点打磨
- 提醒频率需要控制，避免用户被过度打扰

## 19. 已确认关键决策

- 备忘录与待办合并，统一为单一 Record 列表视图
- 标签与分类合并为同一套标签体系
- 周报 / 月报支持用户框选时间范围
- 周报 / 月报支持模板配置与一键生成
- 提醒通知进入 V1
- 左侧菜单保持简洁，不堆叠过多系统入口
- 左侧保留标签快速选择与待办数量展示
- AI 报告、回收站、设置放入左下角按钮区域
- AI 报告保存后默认进入报告历史，不默认污染主列表
- 一键改期支持展开命中任务，并仅调整选中部分
- AI 能力在绑定通道失败时允许自动回退到默认通道

## 20. 我对 TimeAura 的产品建议结论

如果你的目标是做出一个有辨识度、能体现 AI 价值的桌面应用，TimeAura 不应只停留在“更好看的备忘录”，而应该把三个差异化点做深：

- 单行快速记录 + 智能结构化解析
- 完成时间语义化配置
- 即将过期任务一键调整与积压提醒
- 基于真实记录自动生成周报 / 月报 / 复盘

这四点组合起来，才能真正把 slogan“让每个重要时刻如约而至”落到产品体验上。
