# TimeAura Desktop 验收清单

## 1. 当前结论

当前桌面端已经进入一轮可体验验收节点，建议先按“主工作台 -> 提醒与通知 -> AI 通道 -> AI 报告”的顺序做第一轮产品体验。

本轮进入验收的基础条件：

- 桌面壳可编译
- React 页面可跑通
- `src-tauri` 命令级 smoke 测试已补齐
- SQLite 首次启动会自动准备一组演示记录与标签，避免空壳无法体验
- 工作台、提醒、通知、快捷键、标签管理已有覆盖
- AI 通道配置页与 AI 报告页关键交互已补测试
- 当前桌面端前端测试、Tauri Rust 测试与覆盖率检查已通过

---

## 2. 启动方式

### 2.1 Web / 浏览器壳快速体验

在 `apps/desktop` 目录执行：

```bash
npm run dev
```

适合先验证：

- 页面结构
- 列表 / 详情联动
- 标签筛选
- 快速新增
- AI 通道配置页
- AI 报告页

### 2.2 Tauri 桌面壳体验

在 `apps/desktop` 目录执行：

```bash
npm run tauri:dev
```

适合进一步验证：

- 桌面通知
- 通知动作回跳
- Stronghold 凭证写入流程
- SQLite / Tauri 运行时行为

## 3. 第一轮建议验收路径

### 3.1 主工作台

重点看：

- 左侧系统视图切换是否自然
- 标签导航计数是否正确刷新
- 顶部快速新增是否能直接写入列表
- 点击列表项后右侧详情是否按预期展开
- Inspector 风格字段行是否有明显表单感残留

建议操作：

1. 在“全部”视图新建 2 到 3 条记录
2. 分别设置不同优先级、状态、标签、时间
3. 验证中间列表、左侧计数、右侧详情是否同步

### 3.2 提醒条与改期

重点看：

- 提醒条是否能展开命中任务
- 勾选部分任务后“仅改选中”是否符合预期
- 自定义改期时间弹层是否足够清晰
- 一键改期模板是否够快

建议操作：

1. 制造几条即将到期或逾期记录
2. 展开提醒条
3. 分别测试“改全部”“仅改选中”“自定义时间”

### 3.3 通知链路

重点看：

- 桌面通知是否被正确投递
- 通知动作“完成 / 稍后提醒 / 打开详情”是否闭环
- 失败兜底提示是否足够明确
- 通知调试面板日志是否便于排查

建议操作：

1. 在 Tauri 环境触发提醒
2. 依次点击不同通知动作
3. 回到工作台核对状态、时间和定位是否正确

### 3.4 AI 通道配置页

重点看：

- 列表切换与详情编辑是否顺手
- OpenAI Compatible / Anthropic / Azure / Local Gateway / Aggregator 差异字段是否清楚
- API Key 保存 / 清除流程是否自然
- 默认通道、复制、删除、测试连接是否形成闭环
- 能力映射是否容易理解

建议操作：

1. 新建一个通道
2. 切换不同协议观察字段变化
3. 保存凭证、测试连接、设为默认通道
4. 给“AI 摘要 / 周报生成”等能力指定通道

### 3.5 AI 报告页

重点看：

- 时间范围选择是否清晰
- 模板、标签、状态过滤是否容易理解
- 草稿生成、保存历史、存为记录链路是否自然
- 历史记录与预览区是否足够直观

建议操作：

1. 选择周报模板并框选时间范围
2. 生成草稿
3. 保存历史
4. 存为记录并回到工作台核对

## 5. 建议你优先反馈的验收问题

如果开始第一轮体验，最值得优先反馈的是这几类问题：

- 哪个页面最不像真实桌面工具
- 哪个交互最“卡一下”或者不够直接
- 哪个字段命名仍然偏工程化
- 通知链路里哪一步最不安心
- AI 通道配置页里哪些配置仍然难懂
- AI 报告页里哪些步骤应该更短
